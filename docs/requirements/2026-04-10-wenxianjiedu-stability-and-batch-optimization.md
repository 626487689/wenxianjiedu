# 文献解读系统需求文档（2026-04-10）

## 1. 背景

当前项目已具备文档导入、提示词模板、OpenAI 兼容模型调用、单篇解读、批量串行处理、长文分块与降级输出等基础能力，但在批处理稳定性、取消语义、模型适配能力和 AI 运行稳健性方面仍存在明显短板。

## 2. 本次目标

本次按“稳健版”推进，目标如下：

1. 修复批处理取消与状态统计问题。
2. 增强模型适配能力，使 OpenAI-compatible 接口接入更稳健、更可配置。
3. 增强批处理能力，提高真实批量处理场景下的可用性。
4. 对长文解读链路做有限但明确的 AI 稳定性优化。

## 3. 本次交付物

1. 可运行的前端与用例层功能改动。
2. 模型配置与批处理配置的 UI/数据结构更新。
3. 批处理执行逻辑增强。
4. 错误分类与状态展示优化。
5. 必要的验证结果与运行说明。

## 4. 约束

1. 保持单一运行时权威，不引入第二套路由/任务运行框架。
2. 优先复用现有 `appFacade / usecases / services / repositories` 分层。
3. 不承诺一次性支持所有厂商原生协议，本次以 OpenAI-compatible 增强适配为主。
4. 不做云同步、账号体系、数据库、RAG、多窗口。
5. 不基于猜测宣称“AI 质量显著提升”，仅承诺提升稳定性、容错与可配置性。

## 5. 验收标准

### 5.1 Bug 与状态语义

1. 用户取消批处理时：
   - 当前运行项应标记为 `cancelled` 或等价取消态；
   - 未执行项标记为 `cancelled`；
   - 主动取消不得计入 `failed`。
2. 单篇与批量模式的结束态语义一致，仅包含 `success / failed / cancelled`。
3. UI 对取消、超时、接口配置错误、鉴权失败、响应格式异常、解析失败有更清晰区分。

### 5.2 模型适配

1. 用户可明确感知最终请求接口形式。
2. Endpoint 归一化不再只有单一路径拼接策略，需支持自动与显式控制。
3. OpenAI-compatible 接口在 JSON / SSE 等常见返回形式下具有更稳健的解析行为。

### 5.3 批处理能力

1. 支持配置并发数。
2. 支持配置失败重试次数。
3. 支持跳过已有输出。
4. 批处理结束后能够得到汇总结果（UI 或输出报告至少一种）。

### 5.4 AI 稳定性

1. 长文分块后的汇总阶段具备更明确的失败恢复/降级路径。
2. 长文运行过程中可向用户展示更清晰的阶段状态。

## 6. 手动 spot checks

1. 使用一个单文件样例验证单篇解读仍可正常完成。
2. 使用多个文件验证批处理状态统计、取消、失败重试、跳过已有输出。
3. 使用一个长文样例验证分块、汇总、降级输出链路。
4. 使用错误 endpoint / 错误 key / 超短 timeout 验证错误分类与提示文案。

## 7. 非目标

1. 不接入非 OpenAI-compatible 的全新原生协议族。
2. 不实现任务持久化队列。
3. 不新增后端服务。
4. 不做账号、同步、数据库功能。

## 8. 自主性与假设

### 自主性模式

先冻结文档与计划，待用户确认后再实施。

### 推断假设

1. 用户的“可以”表示接受前述推荐的 A 方案（稳健版）。
2. 本次允许同时修改前端 UI、应用编排层、用例层、模型调用层。
3. 本次以可运行改造为主，不强制要求完整自动化测试框架接入。

## 9. 完成语言策略

只有在完成实现、完成验证、并具备证据时，才能使用“已完成”。
在此之前只使用“已冻结需求 / 已输出计划 / 待实施 / 待验证”等表述。

## 10. 交付真实性约束

不得把未验证的推断表述为已支持能力。
如遇模型供应商差异或运行时能力边界，应明确说明而非过度承诺。
